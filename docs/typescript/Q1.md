# TypeScript 面试题文档补充建议

## 总体评价

`Q.md` 的覆盖面比较完整，已经包含基础类型、高级类型、泛型、类型守卫、装饰器、模块声明、工程配置、类型系统陷阱、JS 迁移和编译优化等主题。对于 8 年经验前端来说，题目方向是对的，能覆盖从日常开发到架构治理中的大部分 TypeScript 考点。

不过当前文档主要是“问题清单”，如果用于面试准备或沉淀题库，还可以补充几个维度：

- 增加每类题目的考察重点，避免只背概念。
- 增加典型代码题，让候选人能展示真实类型建模能力。
- 增加工程实践题，尤其是大型前端项目、组件库、monorepo、CI 类型检查等场景。
- 增加追问方向，用来区分“会用”和“理解类型系统边界”。
- 更新部分新版本 TypeScript 特性，例如 `satisfies`、`const` 泛型参数、标准装饰器等。

## 建议调整原文结构

### 1. 按难度或面试阶段分层

当前题目按主题分类，适合作为知识点清单。但如果用于面试，可以进一步分成：

- 初级确认题：基础类型、联合类型、类型注解、接口与类型别名。
- 中级应用题：泛型、类型守卫、工具类型、函数重载、声明文件。
- 高级设计题：类型体操、可辨识联合、类型安全事件系统、SDK/API 类型建模。
- 资深工程题：tsconfig 策略、monorepo、构建性能、类型治理、渐进迁移。

这样更方便根据候选人水平动态调整深度。

### 2. 每道题增加“考察点”

例如：

```md
### `any`、`unknown`、`never`、`void` 的区别？

考察点：

- 是否理解类型安全边界
- 是否知道 `unknown` 需要类型收窄
- 是否能用 `never` 做穷尽检查
- 是否知道 `void` 和 `undefined` 的区别
```

这样面试官或复习者更容易判断回答质量。

### 3. 加入标准答案轮廓

不一定要写完整答案，但建议每题至少补充：

- 关键词
- 示例代码
- 常见误区
- 可继续追问的问题

这会让文档从“题目列表”升级成“可复习、可面试、可评估”的材料。

## 原文中可以优化或细化的题目

### `可选属性` 与 `readonly`

原题问“底层实现差异”，这个说法可以更准确一些。`?` 和 `readonly` 都是类型层面的修饰符，不存在真正的运行时底层实现。建议改成：

> 可选属性 `?` 与只读属性 `readonly` 在类型检查层面分别约束了什么？它们是否会影响运行时？

可追问：

- `readonly` 能否阻止对象内部值被修改？
- `Readonly<T>` 是深只读还是浅只读？
- `exactOptionalPropertyTypes` 开启前后，可选属性的语义有什么区别？

### 装饰器相关题目

原文以“实验性装饰器”为主，但现在需要区分旧版实验性装饰器和新版标准装饰器。建议补充：

- 旧版 `experimentalDecorators` 与标准装饰器语义不同。
- `emitDecoratorMetadata` 主要服务于旧版装饰器生态，例如依赖注入。
- 新标准装饰器并不等价于旧版装饰器，迁移时要谨慎。

### `namespace` 相关题目

可以补一句判断标准：

- 现代前端项目优先使用 ES Module。
- `namespace` 更多出现在历史代码、全局声明、内部类型组织或 `.d.ts` 中。

### `import type` 相关题目

建议补充：

- `import type` 主要影响类型导入在编译产物中的保留或擦除。
- 是否减少最终 bundle 体积，还取决于编译器配置、打包器 tree shaking 和导入对象是否原本只用于类型。

### `skipLibCheck`

建议补充工程判断：

- 优点是提升类型检查速度，绕过第三方声明文件冲突。
- 风险是隐藏依赖包声明错误，尤其在多版本 `@types` 或复杂 monorepo 中。
- 一般可以在应用项目中权衡开启，但在库项目中要更谨慎。

## 建议补充示例答案片段

### `satisfies` 示例

```ts
type Route = {
  path: string;
  method: 'GET' | 'POST';
};

const route = {
  path: '/users',
  method: 'GET',
} satisfies Route;

// route.method 仍然可以保持较精确的字面量推断
```

`satisfies` 适合在“校验对象满足某个结构，同时保留对象自身更精确推断”的场景使用。它比 `as` 更安全，因为 `as` 可能绕过部分类型检查。

### `never` 穷尽检查示例

```ts
type State =
  | { type: 'loading' }
  | { type: 'success'; data: string }
  | { type: 'error'; message: string };

function render(state: State) {
  switch (state.type) {
    case 'loading':
      return 'loading';
    case 'success':
      return state.data;
    case 'error':
      return state.message;
    default: {
      const exhaustive: never = state;
      return exhaustive;
    }
  }
}
```

这类题能考察候选人是否真正理解 `never` 在类型系统中的用途，而不只是知道它表示“不会返回”。
